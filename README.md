## Planner Idea

The planner keeps the existing A* style structure but shapes the search for parking:

- Use obstacle and parking-line distance as risk cost.
- Select entry direction using slot-line semantics, but also evaluate the opposite side when needed.
- Use a local terminal Hybrid A* over `(x, y, yaw, gear)` to produce curved waypoints.
- Add a start stabilizer waypoint so the vehicle does not immediately cut into the first parking-line edge.
- Use asynchronous planning in Pygame mode to avoid IPC timeout.

Core cost form:

```text
c(e) = w_d C_dist(e) + w_r C_risk(e) + w_s C_smooth(e)

R(x, y) = exp(-d_obs(x, y)^2 / (2 sigma^2))

h(e, r) = sqrt(e^T Q(r) e)

Q(r) = Q_far + lambda(r) (Q_near - Q_far)
lambda(r) = exp(-r / sigma_g)
```

Where `e = [x - x_g, y - y_g, wrap(yaw - yaw_g)]^T`, `r` is distance to the goal pose, and `Q_near` gives larger weight to heading/lateral alignment near the parking slot.

## How It Is Implemented

The simulator score function is not modified. The added logic is used inside the planner to choose safer and easier-to-track waypoint candidates before sending control commands.

1. Entry candidates are sampled around the target slot using the slot center, target yaw, and slot size. The semantic side is tried first, but the opposite side is also evaluated if the preview predicts collision.

2. The global approach path is generated with grid A*. Occupied slots and parking lines are treated as blocked regions with safety margins. The risk idea in `R(x, y)` is implemented as clearance-based blocking and penalty:

```text
C_risk ~= 1 / max(clearance, epsilon)
```

This keeps the path away from obstacles and parking lines without changing the professor simulator's collision or score rules.

3. Near the parking slot, terminal Hybrid A* searches over `(x, y, yaw, gear)`. Each edge is generated by a short bicycle-model rollout:

```text
x'     = x + ds cos(theta)
y'     = y + ds sin(theta)
theta' = theta + ds / L * tan(delta)
```

Here `delta` is sampled from left / straight / right steering values, and `gear` is sampled from drive / reverse. This is why the terminal trajectory can contain curved motion rather than only straight line segments.

4. Each full candidate path is previewed with the same kinematic update used by the controller-side tracking approximation. Candidates that collide receive a large penalty; successful candidates are ranked by time, distance, steering flips, gear switches, and grid cost:

```text
S = S_preview + 0.025 C_grid - bonus_semantic + penalty_fallback
```

So the final selected path is not just the shortest path. It is the path that is short enough, collision-free in preview, aligned with the target pose, and reasonably smooth for the controller to follow.

## Implementation Sketch

```python
def compute_path(obs):
    target_pose = get_target_pose(obs["target_slot"])
    semantic_sign = choose_entry_side_from_slot_lines(target_pose)

    candidates = []
    for sign in [semantic_sign, -semantic_sign]:
        global_path = risk_aware_grid_astar(start, entry_pose(sign))
        terminal_path = hybrid_astar(global_path[-1], target_pose)
        candidates.append(preview_and_score(global_path + terminal_path))

    waypoints = min(candidates, key=lambda c: c.score).waypoints
    return waypoints
```
